# 评测器在慢代理下的修复 & 一个 PDF 分页现象 —— 说明与边界

本文档解释我们对 OSWorld **评测器（evaluator getter）** 做的一处改动，以及一个看起来"扣分"
但**我们故意不改**的 PDF 现象。

> ⚠️ **为什么要专门写这份说明**：改动评测/评分相关的代码，天然容易被怀疑是"为了刷分而动评测"。
> 所以这里把**改了什么、为什么这不算篡改评分、什么坚决不改**讲清楚，留作审计依据。
> 相关：评测总流程 [`复现.md`](复现.md)；网络/代理 [`CLAUDE.md`](CLAUDE.md)；记忆 `osworld-evaluator-networkidle`。

---

## 1. 背景：这台机器的网络让评测器"假阴性"

这台宿主**没有直连外网**，VM/评测器访问互联网只能走**高延迟反向代理**（家用主机美国 VPN，
延迟 5–20s、偶发更慢，详见 [`CLAUDE.md`](CLAUDE.md)）。

OSWorld 的多个 **getter**（评测时去抓"参考答案"用的）用 Playwright
`wait_until="networkidle"` 等页面加载。`networkidle` = **页面 load 完 + 连续 500ms 没有任何网络请求**。
问题：

- 在慢代理下，内容多的页面资源要 1–3 分钟才下完，原来的 **60s 超时根本不够**；
- 更要命的是，很多"静态"博客（带 Google Analytics / 评论组件等）**会持续发后台请求**，
  `networkidle` 的"500ms 零连接"**永远满足不了**——实测把超时加到 **180s 仍然超时**。

原代码遇到超时后：抛异常 → 耗尽重试 → **写一个空的 / 垃圾参考文件** → 比对崩溃或恒为 0。
**结果：一个本来做对了的模型，被评测器自己的故障判成 0 分。** 这是**评测基础设施的假阴性**，
不是模型能力问题——正是我们一直要避免的"环境故障被记成模型失败"。

实锤：`chrome/e1e75309-...`（网页转 PDF）模型**完整正确执行**，却因参考 PDF 抓不出来记 **0.00**；
修复后真实分 **0.96**（为什么不是 1.0 见 §4）。

## 2. 改了什么：`networkidle → load → domcontentloaded` 优雅降级

文件：`Env/OSWorld/desktop_env/evaluators/getters/chrome.py`。新增两个共享 helper：

```python
_WAIT_PLAN = (("networkidle", 30000), ("load", 180000), ("domcontentloaded", 60000))
def _goto_with_fallback(page, url, logtag):      # 导航+等待，逐级降级；全失败才抛
def _wait_load_with_fallback(page, logtag):      # 只等待（对已打开的页），尽力而为
```

降级顺序的依据（实测得出）：
- **`networkidle` 给短超时（30s）**：快网络下它几秒就 settle，**上游/别人行为完全不变**；慢代理下
  它注定不 settle，快速失败、不空耗。
- **`load` 给长超时（180s）作主力**：`load` = 所有资源下完即触发（不要求网络静默），这才是
  "页面已完整渲染"的真实信号，给足时间让慢代理把资源下完。
- **`domcontentloaded`（60s）兜底**：极端情况下至少拿到 DOM。

应用到的 **7 处**（6 处其它 getter + 1 处 PDF）：

| 函数 / 用途 | 原 | 现 |
|---|---|---|
| `get_pdf_from_url`（网页转 PDF 的 gold） | networkidle 60s ×3 → 空 PDF | `_goto_with_fallback` |
| `get_page_info`（抓页面信息） | networkidle | `_goto_with_fallback` + `_wait_load_with_fallback` |
| `get_open_tabs_info`（抓标签页） | networkidle | `_wait_load_with_fallback` |
| `get_chrome_saved_address`（`chrome://settings`，本地页） | networkidle | `_goto_with_fallback`（本地页本就秒过，仅为一致） |
| `get_..._search_results`（搜索结果页） | networkidle | `_goto_with_fallback` |
| recreation 取页（已打开页） | networkidle 60s | `_wait_load_with_fallback` |
| recreation 取页（额外等待） | networkidle 30s | `_wait_load_with_fallback` |

> 还有一处（`~2410` 行 recreation.gov 导航）**原本就已经**用 `['domcontentloaded','load','networkidle']`
> 列表逐级尝试，自带降级，无需改。

## 3. 为什么这**不是**篡改评分（边界）

这是本文最重要的一节。判断"改评测是否正当"的标准是：**有没有动评判的内容/阈值，让本不该过的过、
本该过的不过。** 我们的改动**严格只动"怎么等页面加载"，不动"判什么、判多严"**：

| 维度 | 是否改动 |
|---|---|
| 比对什么内容（PDF 文本 / 页面字段 / 文件…） | ❌ 没动 |
| 判定阈值 / `compare_*` 算法 / `conj` 逻辑 | ❌ 没动 |
| 期望答案（expected）/ 评分公式 | ❌ 没动 |
| **等待页面加载用的 load-state 与超时** | ✅ 只动了这个 |
| 快网络（能正常 networkidle）下的行为 | ❌ 不变（首选仍 networkidle，秒过即用） |

换句话说：
- 原行为在我们的网络下会**产出"空参考"→ 恒定假 0**；新行为**产出真实参考**，让比对**正常发生**。
  我们让评测**能正确运行**，而不是让模型更容易过。
- 在**正常网络**上（OSWorld 作者、别人复现的环境）`networkidle` 几秒 settle，走的还是原路径，
  **零影响**。这条改动只在"原本必然假 0"的慢代理场景下生效。
- 这是**纠正评测器的环境脆弱性**，方向是让分数**更接近真实**，不是把分数往上抬。

## 4. 一个故意不改的现象：打印 PDF ≠ 脚本下载 PDF（0.96 而非 1.0）

`chrome/e1e75309-...` 修好后得 **0.96**。为什么不是满分？我们把两个 PDF 的**实际属性**抽出来对比
（`fitz` 读页面尺寸/字体/文本）：

| | 模型"打印"的 PDF | 脚本"下载"的 gold |
|---|---|---|
| 生成方式 | VM 内 Chrome「另存为 PDF」，边距 None（任务要求） | Playwright `page.pdf()`（边距默认也是 None） |
| **纸张尺寸** | **A4** = 595.0×841.9pt (8.26×11.69in) | **US Letter** = 612.0×792.0pt (8.50×11.00in) |
| 页数 | **22 页** | **23 页** |
| 文本字符数 | 43285 | 43287（仅差 2 字符） |
| 开头文本 | 完全一致 | 完全一致 |

**两份是同一篇网页、同样的文字**，根因是**纸张尺寸不同 → 分页点不同**：
- 模型在 VM 里打印，VM 的 Chrome 默认纸张是 **A4**；`page.pdf()` 默认纸张是 **Letter**。
- A4 比 Letter **更高**（841.9pt vs 792pt），每页多塞内容 → 模型 22 页、gold 23 页。
- **边距不是原因**：两边都是无边距（`page.pdf()` 的 `margin` 默认就是全 0；模型也设了 None）。

`compare_pdfs` 用 `fitz` **逐页**抽文本再拼接做模糊匹配（`fuzz.ratio`）。分页点不同 → 词在页边界处的
换行/拼接位置错位 → 同样的词也只得 **0.96**。结尾页脚 `© 2026 Lil'Log Powered by Hugo & PaperMod`
两份都有，只是被分到了不同页。

> 勘误：本节早先版本曾把原因写成"gold 用了 ~0.4 inch 默认边距"——**那是错的**（`page.pdf()` 默认无边距）。
> 实测尺寸证明真正的差异是 **A4(模型) vs Letter(gold)**。

### 为什么我们**不**改 gold（比如强制设成 A4 去对齐模型）

**`page.pdf()` 用默认（Letter、无边距）是 OSWorld 官方评测器的原始行为，官方 75.8% 就是这么测的。**
如果我们为了让分数更高而把 gold 改成 A4 / 别的纸张去迎合模型输出：

- 分数会**偏离官方口径、不可比**；
- 这恰恰是真正的"为刷分而动评测"——**正是我们要避免的**。

所以 **0.96 就是这个任务在官方口径下的忠实分数**，那 0.04 是"A4 vs Letter 分页错位"的固有噪声，
记录在案、不修。

> 对比 §2 和 §4 的取舍，能看清我们的原则：
> **评测器因环境而"无法正常运行"→ 修（让它能跑出真实分）；评测器"正常运行只是不完美"→ 不动（保持与官方一致）。**

---

## 5. 一句话总结

慢代理放大了 OSWorld 评测器里一批 `networkidle` 假阴性（把对的模型判成 0）。我们**只改"怎么等加载"
（降级 + 调超时），不碰任何评判内容/阈值**，且只在原本必然假 0 的场景生效、快网络零影响。
而像 PDF 默认边距这种"和官方一致、只是不完美"的地方，**坚决不动**，以保持分数与官方 75.8% 可比。
