# 复现 Holo-3.1-4B 在 OSWorld-Verified 的结果（人类版）

本文档面向人类操作者，记录在本机（8×RTX3090、rootless docker、反向代理出网）从零复现
Holo-3.1-4B 在 OSWorld-Verified（369 任务）评测的完整步骤，以及踩过的坑。

> 配套还有一份给 AI 代理看的 [`复现_LLM.md`](复现_LLM.md)（命令更密、含故障自愈 playbook）。

## 0. 架构总览（先理解再操作）

本仓库把 **Agent 端**和 **Env 端**解耦：

```
CUA/
  Agent/                    # Agent 端：所有 Holo 相关（与 Env 平级）
    holo_agent/
      agent.py              # HoloComputerAgent：通用官方循环
      tools.py              # Holo 工具集 + Step{note,thought,tool_call} 结构化输出
      recorder.py           # 通用轨迹记录（traj.jsonl + 观察截图 + system_prompt）
      report.py             # 把一次轨迹渲染成自包含 HTML 回放（python -m holo_agent.report）
    docs/holo_official/     # Holo 官方文档（动作/输出规范的唯一权威来源）
  Env/                      # Env 端：通用契约 + 具体环境
    computer_env/           # ComputerEnv 抽象 + InputEvent + StepResult（零 Holo 知识）
    RealtimeGym/            # 一个具体环境（游戏，GameComputerEnv 适配器）
    OSWorld/                # 一个具体环境（桌面）
      holo_repro/           # OSWorld 适配 + 运行器（本次复现的入口）
        osworld_computer_env.py   # OSWorldComputerEnv：tool_call → pyautogui
        run_holo.py               # 组装层：建 DesktopEnv + 跑 Agent + 评测（支持并行）
        prefetch_cache.py         # 预下载所有 setup 输入文件到 cache（离线化）
```

依赖单向：`holo_agent → computer_env`；适配器 `→ computer_env`；`run_holo`（组合根）→ 两者。
**加任何功能前先想清楚放 Agent 端还是 Env 端。**

## 1. 前置条件

- 机器：8×RTX3090（24G），≥1TB 内存（并行跑很多 VM），`/dev/kvm` 存在。
- 模型权重：`/mnt/zhaorunsong/models/Hcompany/Holo-3.1-4B`。
- conda：
  - `vllm_Holo`（vllm 0.23）——跑 vLLM 服务。
  - `osworld`（python 3.12 + OSWorld 全部依赖）——跑评测。

## 3. Python 环境

```bash

```

## 4. 用系统 docker + KVM（关键）

本机 **rootless docker 用不了 KVM**（容器 root 映射成普通用户，`/dev/kvm` 不可写；给用户级 docker 服务加 `SupplementaryGroups=kvm` 会 216/GROUP 失败）。改用**系统级 root docker**：

```bash
export DOCKER_HOST=unix:///var/run/docker.sock   # 当前用户在 docker 组，可访问
export OSWORLD_FORCE_KVM=1                        # 强制启用 KVM（容器是 root，能写 /dev/kvm）
```
provider 已改：`OSWORLD_FORCE_KVM=1` 时挂 `/dev/kvm`，并总是挂 `/dev/net/tun`（否则端口转发失效、Flask:5000 连不上、VM 永远 not ready）。KVM 下 VM 冷启 ~25s（无 KVM 的 TCG ~80s+）。

## 5. 网络 / HF 文件下载（最容易踩坑）

很多任务 setup 要从 `huggingface.co/.../ubuntu_osworld_file_cache` 下载输入文件，**评测阶段**还会下参考文件。HF 直连不通，统一**经反向代理 `127.0.0.1:7897`** 出网。处理：

1. 确认反代可用（应返回 200，反代会跟随 HF 的 LFS/xethub 重定向）：
   ```bash
   https_proxy=http://127.0.0.1:7897 curl -sIL -o /dev/null -w '%{http_code}\n' --max-time 15 \
     "https://huggingface.co/datasets/xlangai/ubuntu_osworld_file_cache/resolve/main/gimp/045bf3ff-9077-4b86-b483-a1040a949cff/gate.jpeg?download=true"
   ```
2. **HF 全部经反代直连 huggingface.co（不要用 hf-mirror！）**：
   - `http_proxy=https_proxy=http://127.0.0.1:7897`，`no_proxy=localhost,127.0.0.1`（vLLM/docker/VM 回环直连不受影响）。反代能连 `huggingface.co` 并跟随其 LFS/xethub 重定向，**大小文件都能下**（实测大图 200）。
   - **不要设 `OSWORLD_HF_MIRROR`**：`hf-mirror.com` 只直供小文件，**把大文件(LFS)302 重定向到不可达的 `cas-bridge.xethub.hf.co`→大文件全失败**（gimp 图、multi_apps 资源等）。`run_holo.py` 里那个"huggingface→mirror 全局补丁"是 `OSWORLD_HF_MIRROR` 开关控制的，**保持不设即关闭**，所有请求直奔 huggingface.co 经反代。
   - 评测期 getter（动态 URL）同样经反代直连 huggingface.co，无需特殊处理。
3. **预下载（推荐）**：先把所有 setup 输入文件下到 `cache/`，运行时命中缓存、几乎不依赖实时网络：
   ```bash
   cd Env/OSWorld && python holo_repro/prefetch_cache.py   # 经反代下 setup 文件到 cache/
   ```
   注：评测期 getter 文件没法可靠预缓存（缓存键各异），运行时仍走反代。

> 经验：反代稳定即可，**别用 hf-mirror**（大文件 LFS 会失败，见上）。`cache/` 预下载到位后运行时基本不依赖实时网络。

## 6. 起 vLLM 服务（4B，可多实例并行）

在 `vllm_Holo` 环境，每个 GPU 起一个 4B 实例（用空闲 GPU，避开他人占用的；GPU0 有掉卡史，谨慎）。例如 GPU 1/2/3/5/7 → 端口 8002-8006：

```bash
ENV=/mnt/zhaorunsong/anaconda3/envs/vllm_Holo
for gpu_port in "1 8002" "2 8003" "3 8004" "5 8005" "7 8006"; do
  set -- $gpu_port
  CUDA_VISIBLE_DEVICES=$1 PATH="$ENV/bin:$PATH" no_proxy=localhost,127.0.0.1 \
  $ENV/bin/python -m vllm.entrypoints.openai.api_server \
    --model /mnt/zhaorunsong/models/Hcompany/Holo-3.1-4B --served-model-name Holo-3.1-4B \
    --tensor-parallel-size 1 --dtype bfloat16 --max-model-len 32768 \
    --max-num-seqs 32 --gpu-memory-utilization 0.90 \
    --limit-mm-per-prompt '{"image": 5}' --trust-remote-code --host 0.0.0.0 --port $2 &
done
# 等各端口 /v1/models 返回 200
```

## 7. 跑评测（并行）

```bash
cd Env/OSWorld
URLS="http://127.0.0.1:8002/v1,http://127.0.0.1:8003/v1,http://127.0.0.1:8004/v1,http://127.0.0.1:8005/v1,http://127.0.0.1:8006/v1"
DOCKER_HOST=unix:///var/run/docker.sock OSWORLD_FORCE_KVM=1 \
  http_proxy=http://127.0.0.1:7897 https_proxy=http://127.0.0.1:7897 \
  no_proxy=localhost,127.0.0.1 \
  python holo_repro/run_holo.py --workers 10 --base_urls "$URLS" \
  > full_eval.log 2>&1 &
# 注意：不要设 OSWORLD_HF_MIRROR（hf-mirror 会坑大文件，见 §5）；经反代直连 huggingface.co 即可
```
- 默认 `--max_steps 100`、全 369 任务、**断点续跑**（跳过已有 `result.txt` 的）、每任务自动生成 `report.html`。
- 10 个 worker（多进程，每 worker 独立 VM/容器），轮询 5 个服务。
- 结果在 `results/pyautogui/screenshot/Holo-3.1-4B/<domain>/<id>/`。

## 9. 停止评测（务必干净）

`pkill -f run_holo.py` **只杀主进程，杀不到 spawn 出的 worker 子进程**（孤儿会继续跑、污染结果）。必须：
```bash
pkill -9 -f "holo_repro/run_holo.py"
for p in $(ps -eo pid,cmd|awk '/envs\/osworld\/bin\/python -c/ && /multiprocessing.spawn/{print $1}'); do kill -9 $p; done
# 清容器
DOCKER_HOST=unix:///var/run/docker.sock docker ps -aq --filter ancestor=happysixd/osworld-docker | xargs -r docker rm -f
```
> 注意别误杀同机其他用户（如 lxy 的 vllm084 也用 multiprocessing.spawn）——按 `envs/osworld` 路径过滤。

## 10. 已知坑 & 注意事项（排查低分时重点看）

- **rootless docker 无 KVM** → 必须系统 docker + `OSWORLD_FORCE_KVM=1`（§4）。
- **HF 直连不通** → 经反向代理 `127.0.0.1:7897` 直连 huggingface.co（§5）。**别用 hf-mirror**：它把大文件 LFS 重定向到不可达的 xethub CDN → 大文件全失败。反代能跟随重定向、大小文件都行。
- **反代偶尔不稳**：反代一断，需下载/联网的任务会崩或被误判（0 分）。建议预下载 setup 文件到 `cache/`（§5.3），并监控 `huggingface.co.*timed out`/`ProxyError`/`Failed to download`。
- **VM 访问真实网站**（chrome 域等）：需 VM 能出网。`run_holo.py` 用 `enable_proxy=True` 让 VM 经宿主反代（tinyproxy 桥接到 7897）访问外网；反代一断这类任务会失败。
- **孤儿 worker**（§9）：停评测没杀干净 → 旧代码/旧环境的 worker 继续写结果，**污染数据**。换配置后务必清空 `results/` 重跑。
- **docker 端口分配锁**：并行启动多 worker 抢 `/tmp/docker_port_allocation.lck`，已把 provider `LOCK_TIMEOUT` 调到 180、worker 启动错峰 8s。
- **分数偏低（当前 ~0.46，官方 4B=75.8%）的原因（诊断仍在进行，结论未定，详见 `复现_debug_plan.md`）**：
  > ⚠️ 别下"主因是 Agent 行为"这类绝对结论——当前工作是**先排除网络/环境/权限等影响因素**。早先写过"空轨迹假 0 极少/网络故障窗口误判 0 已排除/漏保存是最大可修因"等带具体数字的聚合结论，那是没跑遍并逐个核对全部样本就写下的，已作废。
  待排查的候选（并列，未定权重）：
  1. **环境/权限/网络**：VM geo/代理是否每任务真生效、setup 下载与评测期 getter 是否偶发失败被记成 0、慢代理下评测器假阴性、权限类任务、需登录的 gdrive 任务——这些 0 **不是模型能力问题**，要分清后才能统计真实分布；
  2. `max_steps`、`temperature`、工具集/提示词是否与官方设置一致（官方未公布 system prompt，正在做 v1/v2 提示词 A/B）；
  3. `write` 工具用 `pyautogui.write`，非 ASCII 文本不稳；
  4. 坐标缩放、observation 图像预算（最近 3 张）、终止判定（`answer`）等是否符合官方 agent-loop。
  逐任务用 §8 的 `report.html` 核对模型「所见/所想/所做」、把「环境/权限失败」与「模型失败」分清，是定位问题的最快方式。
