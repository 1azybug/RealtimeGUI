# 作废环境失败结果 & 重跑 —— 操作手册（人类版）

本文档记录在多采样评测（`run_multi.py`）跑出的结果里，**把"因环境/网络问题而失败"的格子作废、并让它们重跑**的完整步骤。

> 背景：家用主机供网的反向代理（7897）会**偶尔断**。断网窗口里跑的 web 任务（chrome/multi_apps）
> 会失败，但这**不是模型能力问题**，不该计成 0 分。本手册教你把这类结果挑出来、作废、重跑。
> 评测总流程见 [`复现.md`](复现.md) / [`复现_LLM.md`](复现_LLM.md)；网络/geo 修复见 `CLAUDE.md`。

---

## 0. 核心机制（先理解）

- 结果按 **(任务, prompt, attempt)** 存成一个个"格子"目录：
  ```
  Env/OSWorld/results_multi/<prompt>/<attempt>/pyautogui/screenshot/Holo-3.1-4B/<domain>/<id>/
      ├── result.txt      ← 0/1 分（有它=这格"已完成"）
      ├── traj.jsonl       ← 逐步轨迹（模型所见所想所做）
      ├── step_*.png       ← 每步观察截图
      ├── meta.json        ← 分数 / net_ok / prompt 变体 等
      └── report.html      ← 可视化回放（默认不生成，需手动补，见 §2）
  ```
- **续跑规则**：`run_holo` 看到 `result.txt` 就**跳过**这一格。
- 所以 **"作废" = 删掉那一格目录**（连 `result.txt` 一起）→ 下次 `run_multi`（resume）会**重跑这一格**。

三种判定"环境失败"的途径，从易到准：
| 途径 | 怎么用 | 适用 |
|---|---|---|
| **人工看 report.html**（最可靠） | §2–§4 | 任何结果；程序分不清模型 vs 网络时靠人眼 |
| **网络故障标记 `--net-fail`** | §5 | **只对"全程网络监控生效后新跑的"结果准**（见 §5 说明） |
| **时间窗口 + 限 web 域** | §6 | 你知道某段时间主机断网时，批量作废那段的 web 任务 |

## 2. 给候选任务补生成 report.html

多采样为省磁盘带 `--no_report` 跑，结果里**没有 report.html**（但 traj+截图都在，可随时补生成）。

```bash

$PYO holo_repro/gen_reports.py --domains chrome multi_apps --zero-only

# 也可只给某些任务 / 某些 attempt 生成：
$PYO holo_repro/gen_reports.py --tasks chrome/fc6d8143-9452-4171-9459-7f515143419a
$PYO holo_repro/gen_reports.py --domains chrome --attempts v1/a1 v2/a1 --zero-only
```
列出生成的 report 路径：
```bash
find results_multi -name report.html | sort | head -50
```

## 3. 人工判断：模型问题 还是 网络问题

浏览器打开 report.html，看每步的**观察截图**：

- **网络/环境问题 → 作废**：截图出现
  - `ERR_CONNECTION_RESET` / `ERR_EMPTY_RESPONSE` / `This site can't be reached`
  - `ERR_PROXY_CONNECTION_FAILED` / `Proxy Configuration Help`（代理没通）
  - 中国区弹窗（geo 没生效，说明那一刻 VM 没走美国代理）
- **模型问题 → 保留**（这是真实能力，记 0 是对的）：
  - 页面正常加载，但模型**点错地方 / 计划错 / 没做完 / 死循环 / 没存盘**。

> 经验：report.html 顶部有 SUCCESS/FAIL 状态条；橙色准星标出模型点击处，便于看它"看到了什么、点了哪"。

## 4. 手动作废（删目录）+ 重跑

**作废 = 删掉那一格的整个任务目录**（删到 `<id>` 那层）：
```bash
rm -rf results_multi/v1/a1/pyautogui/screenshot/Holo-3.1-4B/chrome/fc6d8143-9452-4171-9459-7f515143419a
```
> 删错不要紧——删的只是这一次结果，重跑会重新生成；但**别删错成模型失败的**（那会把真实 0 分也重跑、浪费时间）。

**重跑被作废的格子**（resume 自动补跑缺 `result.txt` 的格子）：
```bash
# 1) 确认 geo 路通：宿主回环口 + VM 网桥口都该出美国（由家用 SSH 反向隧道供网，不需要本地中继）：
curl -x http://127.0.0.1:7897   'http://ip-api.com/json?fields=country,city'   # 宿主：应 United States / Los Angeles
curl -x http://10.200.0.1:7897 'http://ip-api.com/json?fields=country,city'   # VM 网桥口：同上；若不通=家用隧道断了，去家用主机重连，别在服务器起中继
# 2) 带原 env 重启 run_multi(resume)：
HOLO_VM_PROXY=http://10.200.0.1:7897 HOLO_NET_GATE=1 \
  $PYO holo_repro/run_multi.py \
  --base_urls "http://127.0.0.1:8002/v1,http://127.0.0.1:8003/v1,http://127.0.0.1:8004/v1,http://127.0.0.1:8005/v1" \
  --workers 14
```
> 注意：若当前还有 `run_multi` 在跑、且它**还没轮到**你删的那一格所在的 pass，它会在后续轮次补跑；
> 想**立刻**补跑就先停掉当前再重启（停法见 [`复现_LLM.md`](复现_LLM.md) STEP 7，务必杀干净 spawn worker）。

## 5. （可选，自动）按网络故障标记批量作废 `--net-fail`

`run_holo` **全程监控网络**（每 8s 探一次中继这条 VM 实路），任务期间断过就把 `net_dropped_during=True`、
`net_ok=False` 记进 `meta.json`。据此自动作废：
```bash
$PYO holo_repro/invalidate.py --net-fail --dry-run    # 先预览
$PYO holo_repro/invalidate.py --net-fail              # 真删
```
⚠️ **只对"全程监控生效之后新跑的"结果准**。更早跑的结果，`meta.json` 只探了开头一次（不可靠），
`--net-fail` 抓不全——那些用 §6 的时间窗口，或 §2–§4 人工看。

---

## 6. （可选，批量）按时间窗口 + 限 web 域作废

你知道"主机网络 16:40–16:50 断了一下"，作废**那段时间完成的 web 任务**（务必加 `--domains`，
否则会误删不需要网络的离线任务 os/vlc/libreoffice 等）：
```bash
$PYO holo_repro/invalidate.py --since "16:40" --until "16:50" \
    --domains chrome multi_apps --zero-only --dry-run     # 预览
# 确认无误后去掉 --dry-run 真删
```
时间可写 `HH:MM`（默认今天）或 `YYYY-MM-DD HH:MM`。然后按 §4 重跑。

## 7. 三层准确性保障（理解为什么这样设计）

1. **网络门控 `HOLO_NET_GATE=1`**：每任务开跑前要求**连续探通**才放行 + **全程监控**，断网就暂停/标记。
2. **多采样 best-of-N**：每任务 v1/v2 各 3 次、失败升 10 次——网络好的窗口里命中一次就拿到正确结果。
3. **本手册的人工作废+重跑**：兜底——程序判不准的，人眼看 report.html 定夺。

> 但网络若**长时间块状断**（如连通块只有 ~20s、任务要跑几分钟），multi-sampling 也救不了——
> 这种情况先把隧道弄稳到分钟级，再重跑 web 任务。
